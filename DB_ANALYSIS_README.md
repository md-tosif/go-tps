# Database Analysis CSV - Column Descriptions

This document explains all the columns in the `db_summary.csv` file generated by `analyze_multi_db.py`.

## Quick Summary

The CSV contains comprehensive statistics from multiple go-tps databases, including:
- Transaction counts and success rates
- Submission and confirmation latencies
- TPS (Transactions Per Second) metrics
- Gas usage and costs
- Gas price analysis

---

## Column Descriptions

### Basic Information

| Column | Description |
|--------|-------------|
| `database_name` | Name of the database file (e.g., load-1.db) |
| `db_path` | Full path to the database file |

### Transaction Counts

| Column | Description |
|--------|-------------|
| `total_transactions` | Total number of transactions in the database |
| `successful_transactions` | Number of transactions with status='success' |
| `failed_transactions` | Number of transactions with status='failed' |
| `pending_transactions` | Number of transactions still pending |
| `success_rate_percent` | Percentage of successful transactions |
| `failure_rate_percent` | Percentage of failed transactions |

### Submission Metrics

| Column | Description |
|--------|-------------|
| `submission_window_seconds` | Time window between first and last transaction submission (seconds) |
| `first_submission_time` | Timestamp of first transaction submission |
| `last_submission_time` | Timestamp of last transaction submission |

### Confirmation Latency

**Note:** Confirmation latency is the time from transaction submission to confirmation.

| Column | Description |
|--------|-------------|
| `min_confirmation_latency_ms` | Minimum confirmation time in milliseconds |
| `max_confirmation_latency_ms` | Maximum confirmation time in milliseconds |
| `avg_confirmation_latency_ms` | Average confirmation time in milliseconds |
| `min_confirmation_latency_sec` | Minimum confirmation time in seconds |
| `max_confirmation_latency_sec` | Maximum confirmation time in seconds |
| `avg_confirmation_latency_sec` | Average confirmation time in seconds |

### TPS (Transactions Per Second) Metrics

| Column | Description |
|--------|-------------|
| `tps_submission` | Overall TPS based on submission time window |
| `tps_confirmation` | Overall TPS based on confirmation time window (successful txns only) |
| `confirmation_window_seconds` | Time between first and last confirmation |
| `min_tps_batch` | Minimum TPS across all batches |
| `max_tps_batch` | Maximum TPS across all batches |
| `avg_tps_batch` | Average TPS across all batches |

### Gas Price (Wei)

**Note:** Gas price is the price per unit of gas in wei (1 ETH = 10^18 wei).

| Column | Description |
|--------|-------------|
| `min_gas_price_wei` | Minimum gas price in wei |
| `max_gas_price_wei` | Maximum gas price in wei |
| `avg_gas_price_wei` | Average gas price in wei |

### Gas Price (Gwei)

**Note:** Gwei is a common denomination (1 Gwei = 10^9 wei = 0.000000001 ETH).

| Column | Description |
|--------|-------------|
| `min_gas_price_gwei` | Minimum gas price in Gwei |
| `max_gas_price_gwei` | Maximum gas price in Gwei |
| `avg_gas_price_gwei` | Average gas price in Gwei |

### Effective Gas Price (Wei)

**Note:** Effective gas price is the actual price paid per unit of gas (for EIP-1559 transactions).

| Column | Description |
|--------|-------------|
| `min_effective_gas_price_wei` | Minimum effective gas price in wei |
| `max_effective_gas_price_wei` | Maximum effective gas price in wei |
| `avg_effective_gas_price_wei` | Average effective gas price in wei |

### Effective Gas Price (Gwei)

| Column | Description |
|--------|-------------|
| `min_effective_gas_price_gwei` | Minimum effective gas price in Gwei |
| `max_effective_gas_price_gwei` | Maximum effective gas price in Gwei |
| `avg_effective_gas_price_gwei` | Average effective gas price in Gwei |

### Gas Used

**Note:** Gas used is the actual amount of gas consumed by the transaction.

| Column | Description |
|--------|-------------|
| `min_gas_used` | Minimum gas units consumed per transaction |
| `max_gas_used` | Maximum gas units consumed per transaction |
| `avg_gas_used` | Average gas units consumed per transaction |
| `total_gas_used` | Total gas units consumed by all transactions |

### Gas Limit

**Note:** Gas limit is the maximum gas allowed for the transaction.

| Column | Description |
|--------|-------------|
| `min_gas_limit` | Minimum gas limit set |
| `max_gas_limit` | Maximum gas limit set |
| `avg_gas_limit` | Average gas limit set |

### Transaction Costs

**Note:** Cost = gas_used × effective_gas_price

| Column | Description |
|--------|-------------|
| `total_cost_wei` | Total cost of all transactions in wei |
| `total_cost_eth` | Total cost of all transactions in ETH |
| `avg_cost_per_tx_wei` | Average cost per transaction in wei |
| `avg_cost_per_tx_eth` | Average cost per transaction in ETH |

### Additional Information

| Column | Description |
|--------|-------------|
| `total_batches` | Total number of unique batch runs |
| `unique_wallets` | Number of unique wallet addresses used |

## Usage Examples

### Analyze specific databases
```bash
python3 analyze_multi_db.py load-1.db load-2.db
```

### Analyze all .db files
```bash
python3 analyze_multi_db.py *.db
```

### Custom output file
```bash
python3 analyze_multi_db.py -o my_results.csv load-1.db load-2.db transactions.db
```

```bash
column -s, -t db_summary.csv | less -S
```

Just double-click the `db_summary.csv` file or:
```bash
libreoffice db_summary.csv
```

## Understanding the Metrics

### Latency vs TPS
- **Latency** measures how long each transaction takes
- **TPS** measures throughput (transactions per second)
- Low latency + high TPS = optimal performance

### Gas Price vs Effective Gas Price
- **Gas Price:** The price you're willing to pay
- **Effective Gas Price:** The actual price paid (relevant for EIP-1559)
- For legacy transactions, these are typically the same

### Gas Used vs Gas Limit
- **Gas Limit:** Maximum gas you authorize
- **Gas Used:** Actual gas consumed
- Unused gas is refunded

### Success Rate
- 100% success rate indicates all transactions confirmed
- Lower rates may indicate:
  - Insufficient funds
  - Network congestion
  - RPC endpoint issues
  - Pending transactions not yet confirmed

## Tips for Analysis

1. **Compare TPS across databases** to see performance differences
2. **Check success rates** to identify reliability issues
3. **Monitor gas prices** to track network costs
4. **Analyze confirmation latency** to understand network speed
5. **Review batch statistics** (min/max/avg TPS) to spot performance patterns

## Troubleshooting

### Database not found
Ensure the database file exists in the current directory or provide full path:
```bash
python3 analyze_multi_db.py /path/to/database.db
```

### Empty or zero values
This can happen if:
- Database has no successful transactions
- Transactions are still pending
- Database structure is different than expected

### Very high latency
Check if:
- Network was congested during test
- RPC endpoint was slow
- WebSocket connection was not used
